### 2.2 Core 模块 (core)

#### 2.2.1 RAG 核心抽象

##### AbstractRagFlow (模板方法)

```java
public abstract class AbstractRagFlow implements RagFlow {
    // 子类只需配置这两个方法
    protected abstract void configurePipeline(RagPipeline pipeline);
    protected abstract void configureOrchestrator(RagOrchestrator orchestrator);

    // executeRag 执行骨架
    public RagAnswer executeRag(String question, String userId, String source) {
        RagContext context = createContext(question, userId, source);
        orchestrator.beforeExecute(context);     // 缓存检查 → 记忆 → 复杂度
        if (isCacheHit(context)) return buildCachedAnswer(context);
        RagAnswer answer = pipeline.execute(context); // 管道执行
        orchestrator.afterExecute(context, answer);   // 记忆 → 评估 → 缓存
        return answer;
    }
}
```

#### 2.2.3 管道框架

##### RagPipeline

```java
public interface RagPipeline {
    RagAnswer execute(RagContext context);
    RagPipeline addStage(PipelineStage stage);
}
```

##### DefaultRagPipeline

```java
public class DefaultRagPipeline implements RagPipeline {
    private final List<PipelineStage> stages = new ArrayList<>();

    public RagAnswer execute(RagContext context) {
        for (PipelineStage stage : stages) {
            if (stage.shouldSkip(context)) continue;
            long start = System.currentTimeMillis();
            try {
                stage.process(context);
                context.recordStageDuration(stage.getName(), System.currentTimeMillis() - start);
            } catch (Exception e) {
                // 检索失败 → 空文档继续; 生成失败 → 降级消息
                handleStageFailure(context, stage, e);
            }
        }
        return buildAnswer(context);
    }
}
```

##### PipelineStage

```java
public interface PipelineStage {
    void process(RagContext context);
    String getName();
    default boolean shouldSkip(RagContext context) { return false; }
}
```

#### 2.2.4 管道阶段详解

| 阶段 | 类名 | 职责 | 关键逻辑 |
|------|------|------|---------|
| 1 | `QueryCleaningStage` | 查询清洗 | 去首尾空格、规范化空白字符 |
| 2 | `QueryEnhancementStage` | 查询增强 | 遍历策略列表，依次增强查询。记忆增强 → 关键词扩展 |
| 3 | `MultiQueryGenerationStage` | 多查询生成 | LLM 生成 3 个角度变体，过滤低质量查询 |
| 4 | `ParallelRetrievalStage` | 并行检索 | CompletableFuture 并行执行多查询，3s/查询超时，总 10s 超时 |
| 5 | `RetrievalStage` | 单次检索 | 通过 RetrievalStrategy 执行检索，记录指标 |
| 6 | `CragStage` | 纠错 RAG | LLM 评估检索质量：CORRECT → 保留；AMBIGUOUS → KnowledgeRefiner 精炼重查；INCORRECT → ExternalSearchService |
| 7 | `DeduplicationStage` | 去重 | 内容哈希精确去重 → Jaccard 相似度模糊去重 (>0.8) |
| 8 | `FilteringStage` | 过滤 | 移除 text 长度 < 20 的文档 |
| 9 | `ReRankingStage` | MMR 重排序 | MMR 算法平衡相关性与多样性，lambda 可配置 |
| 10 | `CompressionStage` | 压缩 | 总字符 < 5000 跳过；Embedding 粗压缩 (相似度 >0.6) → LLM 精压缩 |
| 11 | `GenerationStage` | 生成 | 构建系统 Prompt（含引用标记要求）→ LLM 调用（ResilienceHelper 容错）→ 正则提取引用 `[n]` |

#### 2.2.5 检索系统

##### ContentRetriever 接口

```java
public interface ContentRetriever extends State {
    List<Document> retrieve(String query, String source);
    List<Document> retrieve(String query, int topK, double similarityThreshold, String source);
}
```

##### BasicContentRetriever — 纯向量检索

注入 `VectorStore` (Milvus)，构建 `SearchRequest` 执行 `similaritySearch`。

##### HybirdContentRetriever — 混合检索 (@Primary)

```java
retrieve(query, topK, threshold, source) {
    // 1. 向量检索 (topK * 2)
    List<Document> vectorDocs = vectorSearch(query, topK * 2, threshold, source);
    // 2. BM25 关键词检索 (topK * 2)
    List<ScoredDocument> bm25Docs = bm25Indexer.search(query, topK * 2);
    // 3. RRF 融合
    return bm25Docs.isEmpty() ? vectorDocs : mergeWithRrf(vectorDocs, bm25Docs, topK);
}
```

**RRF 融合公式**:
```
RRF(d) = w_vec / (k + r_vec(d)) + w_bm25 / (k + r_bm25(d))
```
- k = 60 (RRF 常数)
- w_vec = 0.7 (向量权重，可配置)
- w_bm25 = 0.3 (BM25 权重，可配置)
- 以文档文本内容 (content) 作为匹配 Key

##### Bm25Indexer — BM25 搜索

**核心参数**: k1 = 1.2, b = 0.75

**数据结构**:
- 倒排索引: `Map<term, Map<docKey, tf>>`
- 文档存储: `Map<docKey, Document>`
- 读写锁: `ReentrantReadWriteLock`

**分词策略**:
- 中文/CJK: 单字切分
- 英文/拉丁: 按空格/标点分割 → 小写 → 过滤单字母
- 数字: 保留

**IDF 公式**: `ln((N - n_t + 0.5) / (n_t + 0.5) + 1)`

**BM25 评分**: `Σ IDF(t) · TF(t,d) · (k1+1) / (TF(t,d) + k1 · (1-b + b·|d|/avgdl))`

##### RetrievalOptimizer — 查询优化

基于 LLM 的三种优化：
1. **查询扩展** (`expand`) — 补充语义相关术语
2. **查询重写** (`rewrite`) — 修正语法和措辞
3. **关键词提取** (`extractKeywords`) — 提取核心关键词

#### 2.2.6 压缩系统

##### HybridCompressor — 两级压缩

1. **EmbeddingCompressor** (粗压缩):
   - `SentencesSplitter` 将文档拆为句子
   - 计算每个句子与查询的余弦相似度
   - 保留相似度 > 阈值 (0.6) 的句子
   - 最少保留 2 个句子

2. **LLMCompressor** (精压缩):
   - 使用 Prompt 提取与问题相关的信息
   - 跳过过长文档

#### 2.2.7 MMR 重排序

```
MMR = argmax [ λ · Sim(Di, Q) - (1-λ) · max Sim(Di, Dj) ]
```

- λ = 0.5 (可配置)
- 余弦相似度通过 EmbeddingModel 计算

#### 2.2.8 记忆管理

##### ShortTermMemoryManager (短期)

- 内存 `ConcurrentHashMap` + Redis 持久化
- 每个会话最多 20 条消息
- TTL 30 分钟 (刷新访问时间时自动续期)
- Key 格式: `session:{userId}`
- 序列化: Jackson JSON (JSR310 模块)

##### LongTermMemoryManager (长期)

- MySQL 持久化 (JPA)
- 三种记忆类型: FACT, PREFERENCE, CONTEXT
- 向量检索: EmbeddingModel 生成向量 → 余弦相似度检索
- 记忆合并: 相似度 ≥ 0.85 时合并 (取高重要性内容 + 去重关键词 + 提升重要性)

##### MemoryExtractor

- 每 10 条消息触发一次 LLM 提取
- 格式: `[FACT|PREFERENCE|CONTEXT] content | keywords | importance`

#### 2.2.9 质量评估

##### RagEvaluator

基于 LLM 的 RAGAS 风格评估，5 个指标:

| 指标 | 评分范围 | 说明 |
|------|---------|------|
| Answer Relevance | 0~1 | 答案与问题的相关性 |
| Faithfulness | 0~1 | 答案对上下文的忠实度 |
| Context Relevance | 0~1 | 上下文与问题的相关性 |
| Context Precision | 0~1 | 上下文中相关信息的比例 |
| Context Recall | 0~1 | 问题所需信息在上下文中的覆盖率 |

总体评分 = 5 项指标平均值。

##### EvaluationManager

- MySQL 持久化评估结果
- 提供用户平均分查询、全局统计、低质量评估查询

##### QualityMonitor

- `@Scheduled(cron = "0 0 3 * * ?")` 每天凌晨 3 点触发
- 识别低分模式和趋势
- 提供改进建议

#### 2.2.16 CRAG 组件

##### RetrievalEvaluator

LLM 评估检索质量，返回 AssessmentEnum:
- `CORRECT` → 直接使用
- `AMBIGUOUS` → KnowledgeRefiner 处理
- `INCORRECT` → ExternalSearchService 搜索

##### ComplexRAGHandler

```java
handle(query, docs) {
    AssessmentEnum assessment = retrievalEvaluator.evaluate(query, docs);
    switch (assessment) {
        case CORRECT:    return docs;
        case AMBIGUOUS:  return knowledgeRefiner.refine(query, docs);
        case INCORRECT:  return externalSearchService.search(query);
    }
}
```

##### KnowledgeRefiner

1. 提取文档中与问题相关的句子 (Embedding 相似度 > 0.5)
2. LLM 重写查询
3. 用重写后的查询再次检索

##### ExternalSearchService

- 支持 Tavily Search API 和 Bing Search API
- 返回 Document 列表 (含 source, title 元数据)
- 异常时静默降级

## 5. 算法详细设计

### 5.1 BM25 评分算法

```
输入: 查询 Q = {t₁, t₂, ..., tₙ}, 文档 Dᵢ
输出: 评分 Score(Q, Dᵢ)

对每个查询词 t ∈ Q:
  n(t)    = 包含 t 的文档数
  IDF(t)  = ln((N - n(t) + 0.5) / (n(t) + 0.5) + 1)
  TF(t,d) = t 在 d 中的出现次数
  |d|     = d 的分词数
  avgdl   = 所有文档的平均长度
  score  += IDF(t) · TF(t,d) · (k₁+1) / (TF(t,d) + k₁ · (1 - b + b · |d|/avgdl))

参数: k₁ = 1.2 (词频饱和度), b = 0.75 (长度归一化)
```

### 5.2 RRF 融合算法

```
输入: 向量排名 V = [d₁, d₂, ..., dₙ], BM25 排名 B = [d₁, d₂, ..., dₘ]
输出: 融合排名

对每个文档 d:
  RRF(d) = w_vec / (k + rank_vec(d)) + w_bm25 / (k + rank_bm25(d))

参数: k = 60, w_vec = 0.7, w_bm25 = 0.3
```

### 5.3 MMR 重排序算法

```
输入: 候选集 C, 查询 Q, λ
输出: 重排序结果 R (初始为空)

while |R| < topK:
  d* = argmax [ λ · Sim(d, Q) - (1-λ) · max_{r∈R} Sim(d, r) ]
  R = R ∪ {d*}
  C = C \ {d*}
  
Sim: 余弦相似度 (cosineSimilarity)
```

### 5.4 问题复杂度分类

```
规则路径: 长度 < 20 且无复杂关键词 → SIMPLE
LLM 路径: Prompt 分类 → SIMPLE / MODERATE / COMPLEX
```

### 5.5 文档分块算法 (RecursiveCharacterTextSplitter)

```
分隔符优先级: \n## → \n### → \n → 。 → ！ → ？ → 空格 → ""
递归切分: 超过 500 字符 → 使用下一优先级分隔符
合并: 相邻片段合并，目标 200~500 字符/块 (末尾残片不强制)
```
