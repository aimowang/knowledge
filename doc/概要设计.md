# RAG 智能知识库系统 — 概要设计

**文档版本**: v1.1 | **编写日期**: 2026-06-14 | **项目名称**: RAG 智能知识库问答系统

## 1. 引言

### 1.1 编写目的
本文档描述 RAG 智能知识库系统的总体架构设计，为详细设计、编码实现和测试提供指导。

### 1.2 项目背景
基于 Spring AI 1.1.7 构建的企业级 RAG（检索增强生成）知识库系统，支持智能问答、记忆管理、质量评估和全方位可观测性。采用管道模式 + 策略模式 + 编排器模式实现灵活的 RAG 流程编排。

## 2. 系统概述

### 2.1 系统目标
1. **智能问答**: 基于混合检索（向量 + BM25）的精准答案生成
2. **记忆管理**: 短期记忆（会话上下文）+ 长期记忆（用户画像）
3. **质量评估**: 自动化评估答案质量（5 项 RAGAS 指标）
4. **企业级特性**: 认证授权、性能优化、可观测性、容错机制

### 2.2 用户角色
| 角色 | 权限 | 说明 |
|------|------|------|
| 普通用户 | 问答、查看历史 | 使用 RAG 问答功能 |
| 管理员 | 用户管理、监控 | 管理系统和查看指标 |

### 3.3 RAG 核心架构

#### 3.3.1 管道模式（Pipeline Pattern）

RAG 流程被拆分为多个独立阶段，按顺序执行。`DefaultRagPipeline` 管理阶段列表，自动记录每个阶段的耗时。

**AdvancedRagFlow 管道**（11 阶段）：

| 序号 | 阶段 | 职责 |
|------|------|------|
| 1 | `QueryCleaningStage` | 去除空格、规范化空白字符 |
| 2 | `QueryEnhancementStage` | 应用增强策略（记忆指代消解 + 关键词扩展） |
| 3 | `MultiQueryGenerationStage` | 复杂问题生成多角度变体查询 |
| 4 | `ParallelRetrievalStage` | 多查询并行向量检索（各 3s 超时，总 10s） |
| 5 | `RetrievalStage` | 单次向量检索（兜底） |
| 6 | `CragStage` | 纠正性 RAG：评估检索质量，模糊/错误时修正 |
| 7 | `DeduplicationStage` | 哈希去重 + Jaccard 相似度去重 (>0.8) |
| 8 | `FilteringStage` | 移除过短文档 (<20 字符) |
| 9 | `ReRankingStage` | MMR 重排序 |
| 10 | `CompressionStage` | Embedding/LLM 两级压缩 |
| 11 | `GenerationStage` | 构建 Prompt、调用 LLM、提取引用 |

**BasicRagFlow 管道**（7 阶段）：同上，但跳过 3-6 和 9。

#### 3.3.2 策略模式（Strategy Pattern）

**检索策略**:
- `DefaultRetrievalStrategy` — 通过 `ContentRetriever` 委托执行检索

**查询增强策略**:
- `MemoryBasedQueryEnhancer` — 基于短期记忆的指代消解
- `KeywordExpansionEnhancer` — 关键词同义词扩展（仅中等/复杂问题）
- `MultiQueryGenerator` — 多角度查询生成

**文档处理策略**:
- `DeduplicationStrategy` — 精确去重（哈希）+ 模糊去重（Jaccard）
- `FilteringStrategy` — 质量过滤（移除短文档）
- `CompressionStrategy` — 上下文压缩（通过 HybridCompressor）

#### 3.3.3 编排器模式（Orchestrator Pattern）

`RagOrchestrator` 负责协调横切关注点，通过功能开关控制：

```
beforeExecute():
  缓存检查 (命中 → 直接返回)
  ↓
  加载短期/长期记忆 (根据开关)
  ↓
  分类问题复杂度 (SIMPLE/MODERATE/COMPLEX)

afterExecute():
  保存短期记忆 → 提取长期记忆 → 异步质量评估 → 缓存结果
```

**功能开关配置**:
| 开关 | BasicRagFlow | AdvancedRagFlow |
|------|-------------|----------------|
| 短期记忆 | ❌ | ✅ |
| 长期记忆 | ❌ | ✅ |
| 质量评估 | ❌ | ✅ |

### 3.5 检索系统

```
┌──────────────────────────────────────────────┐
│              ContentRetriever                 │
│                    ↑                          │
│       ┌────────────┴────────────┐             │
│       ▼                        ▼              │
│  BasicContentRetriever   HybirdContentRetriever│
│  (纯向量检索 @Milvus)     (@Primary 混合检索)   │
│                              ├─ VectorStore   │
│                              └─ Bm25Indexer   │
│                                 (本地 BM25 索引)│
│       RRF(d) = w_vec/(k+r_vec) + w_bm25/(k+r) │
└──────────────────────────────────────────────┘
```

**Bm25Indexer 特性**:
- 经典 BM25 算法 (k1=1.2, b=0.75)
- 中英文混合分词（中文单字、英文按空格/标点分割）
- 倒排索引 + 读写锁线程安全
- 动态添加/移除文档

## 4. 核心业务流程

### 4.1 RAG 问答流程

```
用户请求 (POST /api/qa/ask)
  → JWT 认证 (JwtAuthenticationFilter)
  → 速率限制检查 (RateLimitFilter)
  → KnowledgeQAController.ask()
    → KnowledgeQAService.askInFlowWithSources()
      → LLM 分类问题 (CategoryEnum)
      → 选择 RagFlow (AdvancedRagFlow 优先)
      → AbstractRagFlow.executeRag()
        → RagOrchestrator.beforeExecute()
          缓存检查 → 加载记忆 → 分类复杂度
        → RagPipeline.execute()
          11 阶段顺序执行
        → RagOrchestrator.afterExecute()
          保存记忆 → 提取长期记忆 → 评估 → 缓存
      → 返回 RagAnswer (答案 + 来源)
```

### 4.2 文档嵌入流程

```
上传文件 (DocumentService.upload())
  → 保存文件到磁盘
  → KnowledgeEmbeddingService.embedding()
    → DocumentLoader.load(file)
      PdfLoader / MarkDownLoader / TextLoader
    → ContentSplitter.split(docs)
      RecursiveCharacterTextSplitter (200~500字符/块)
    → VectorStore.add(batch)      — Milvus 向量化存储
    → Bm25Indexer.addDocuments()  — 本地 BM25 索引
  → 写入上传记录到 MySQL (UploadedFile)
```

## 7. 安全设计

### 7.1 认证机制
- **JWT 双 Token**: Access Token (24h) + Refresh Token (7天)
- **密码加密**: BCrypt
- **Token 刷新**: POST `/api/auth/refresh`
- **无状态会话**: SecurityContextHolder 使用 STATELESS 策略

### 7.2 防护措施
- **速率限制**: Bucket4j 按 IP 限流，默认 100 次/分钟
- **SQL 注入**: JPA Parameterized Query
- **CORS**: 全来源开放 (开发环境)
- **端点保护**: `/api/auth/**` 和 `/actuator/**` 公开，其余需认证

## 8. 性能设计

### 8.1 缓存策略

| 缓存 | 存储 | TTL | 说明 |
|------|------|-----|------|
| 问答结果 | Redis | 1 小时 | key: `qa:answer:{userId}:{hash}` |
| 会话消息 | Redis | 30 分钟 | key: `session:{userId}` |
| 查询增强 | Caffeine | 10 分钟 | 关键词/记忆增强缓存 |

### 8.2 线程池隔离

| 线程池 | 核心/最大 | 队列 | 用途 |
|--------|----------|------|------|
| `ragRetrievalExecutor` | 5/10 | 100 | 向量检索 |
| `llmCallExecutor` | 3/6 | 50 | LLM 调用 |

### 8.3 容错配置

| 维度 | LLM 调用 | 向量搜索 |
|------|---------|---------|
| 熔断阈值 | 50% (10 窗口) | 60% (10 窗口) |
| 重试次数 | 3 | 2 |
| 退避策略 | 指数 (2x) | 固定 500ms |
| 超时 | 30 秒 | 10 秒 |

## 9. 可观测性设计

### 9.1 Prometheus 指标
- `rag_cache_hit_total` / `rag_cache_miss_total` — 缓存命中率
- `rag_llm_call_total` — LLM 调用次数
- `rag_vector_search_total` — 向量搜索次数
- `rag_answer_duration_*` — RAG 响应时间直方图
- `rag_llm_duration_*` — LLM 调用时间直方图
- `rag_vector_search_duration_*` — 向量搜索时间直方图

### 9.2 告警规则
1. **ApplicationDown**: 应用下线 > 1 分钟
2. **HighErrorRate**: 5xx 错误率 > 5%
3. **HighLatency**: P95 延迟 > 5 秒
4. **HighMemoryUsage**: JVM 堆内存 > 85%
